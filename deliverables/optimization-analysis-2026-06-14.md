## 一、总览

## 二、数据支持层优化

### 2.1 🔴 Team Profile 数据从 mock 转为真实

**现状**：`data_loader.py` 的 `seed_mock_history()` 用 `hash(team_name)` 生成确定性模拟比赛，所有球队画像的 30+ 指标、3 条调整规则完全基于假数据运行。

**问题**：
- 画像特征的统计特性与真实足球完全脱节（模拟数据的均值/方差/相关结构与真实联赛差异大）
- 调整规则阈值（elo_gap=150/180、resilience=0.58 等）在假数据上验证无意义
- hash 种子的空间均匀性未验证，某些队名可能系统性地产生偏强/偏弱历史
- 16 场模拟比赛支撑 30+ 指标 + 4 级对手分层，很多格子只有 2-4 场样本

**优化方案**：
1. **短期**：接入历史比赛真实数据（FIFA 国际赛历史、世预赛结果等），替换 `seed_mock_history()`
2. **中期**：实现 `as_of` 时间切片的真实数据加载——预测某场时只用该场开赛前可见的数据
3. **长期**：增量更新机制——每场世界杯比赛结束后自动更新画像特征
4. **保护**：保留小样本保护逻辑（`sample_count < 6` 不下强结论），并增加分层样本量展示

**优先级**：🔴 极高——这是当前系统最大的数据质量风险

---

### 2.2 🟠 市场数据源多元化

**现状**：仅从中国体彩（sporttery）获取赔率，使用非官方 Web API 爬虫。

**问题**：
- 无交叉验证：体彩赔率可能受中国市场偏见影响（如亚洲球队赔率偏高/偏低）
- API 稳定性差：非官方接口随时可能因反爬策略变更而失效
- 无历史赔率变化：缺少开盘→临盘的赔率时间序列，而赔率变动本身是重要的信息信号
- 队名映射 `sporttery-team-aliases.json` 手动维护，无自动化测试

**优化方案**：
1. **新增 Bet365/Pinnacle 等国际赔率数据**：作为交叉验证和备选源
2. **增加赔率时间序列**：至少保存开盘赔率和临盘赔率，用于计算赔率变动方向
3. **队名映射自动化测试**：每次赛程更新后自动验证映射完整性
4. **市场数据健康监控**：记录每次获取的成功/失败率，超过阈值时告警

---

### 2.3 🟡 比赛数据实时同步机制

**现状**：比赛赛程来自种子 JSON，比赛结果的数据同步链路不明确。

**问题**：
- 缺少从外部 API（FIFA 官方、数据供应商）自动同步比赛结果的可靠机制
- 比赛进行中的实时比分更新依赖手动触发
- `data/raw/` 目录为空，没有原始数据的持久化

**优化方案**：
1. 接入 football-data.org 或 API-Football 的比赛结果推送/polling
2. 明确数据新鲜度的 SLA（如"赛后 X 分钟内自动更新"）
3. 在工作台展示"数据最后更新时间"和"数据来源"

## 三、预测模型层优化

### 3.1 🔴 Poisson 模型引入 overdispersion

**现状**：`model_configs.yaml` 定义了 `poisson_dispersion` 参数（默认 1.0），但 `poisson.py` 从未读取该值。当前 Poisson 分布的方差恒等于均值，无法表达真实足球比赛中常见的过度离散（方差 > 均值）。

**问题**：
- 标准 Poisson 预测的平局概率系统性偏低（因为方差被低估，两个 Poisson 变量同时取同一值的概率变小）
- 这也是为什么系统需要额外的 `draw_boost` 后处理来弥补
- 7 球截断的尾部概率处理粗糙

**优化方案**：
1. **实现 Negative Binomial 模型**：用 `poisson_dispersion` 参数控制额外方差
   ```python
   # 伪代码
   if dispersion > 1.0:
       # Negative Binomial: mean=λ, variance=λ*dispersion
       r = λ / (dispersion - 1)
       p = 1 / dispersion
       goals_pmf = nbinom.pmf(range(8), r, p)
   else:
       goals_pmf = poisson.pmf(range(8), λ)
   ```
2. **校准 dispersion 值**：用历史比赛数据拟合最优 dispersion 参数
3. **逐步淘汰 draw_boost**：overdispersion 自然产生更高的平局概率，draw_boost 可以回归 1.0

---

### 3.2 🟠 模型参数统计校准

**现状**：8 个模型变体的参数值（draw_boost=1.10/1.15、favorite_dampening=0.10、underdog_boost=0.03 等）均为经验猜测，没有任何超参数搜索或交叉验证。

**问题**：
- 无法量化哪个变体在历史数据上表现最优
- 后处理调整链（draw_boost → favorite_dampening → underdog_boost → upset_factor）依次叠加缺乏理论依据，可能互相放大
- `elo-poisson-v2-calibrated` 这个名字暗示已校准，但实际参数同样是手调

**优化方案**：
1. **引入校准框架**：用过去 2-3 届世界杯（2014/2018/2022）的比赛结果做回测
2. **网格搜索或贝叶斯优化**：对核心参数（draw_boost, str_coeff, base_goal 等）做超参数搜索
3. **记录每个变体的校准历史**：在 `model_configs.yaml` 中增加 `calibration` 字段，记录校准数据集、Brier 分数、样本量
4. **后处理调整链审计**：验证多次调整的联合效应是否产生非预期的概率分布

---

### 3.3 🟡 Elo 模型优化

**现状**：Elo 使用固定 K=30，主场优势固定 +60（中立场地为 0），归一化后丢失绝对尺度。

**问题**：
- K=30 对所有比赛等权，但友谊赛和世界杯淘汰赛的重要性差异巨大
- 主场优势 +60 是 2018 年前的经验值，2026 世界杯在美国/加拿大/墨西哥举行，某些"半主场"效应（如墨西哥在美国比赛）没有被捕捉
- 归一化 `(rating - min) / spread` 使 `str_coeff` 的物理意义变得模糊

**优化方案**：
1. **动态 K 因子**：友谊赛 K=20、小组赛 K=30、淘汰赛 K=40
2. **世界杯特化的主场优势**：墨西哥在美国比赛可能有"半主场"效应（+30 而非 0 或 +60）
3. **保留原始 Elo 尺度**：归一化改为在 `predict_match()` 内部做，保留原始评分的可解释性

---

### 3.4 🟡 淘汰赛模拟校准

**现状**：70% 概率在 90 分钟内决出胜负，加时赛衰减系数 0.3，这两个关键参数无数据支撑。

**问题**：
- 实际上不同淘汰赛阶段（32 强 vs 决赛）平局率差异很大
- 衰减系数 0.3 不区分 30 分钟加时赛和点球大战（后者几乎完全随机）
- `bracket.py` 和 `qualification.py` 各自维护 32 强对阵表，存在代码重复

**优化方案**：
1. **按阶段校准平局率**：32 强 ~25%、16 强 ~30%、半决赛 ~35% 等
2. **区分加时赛和点球大战**：加时赛衰减 0.3-0.5，点球大战接近 0.5（几乎随机）
3. **统一赛程定义**：将 bracket.py 和 qualification.py 的对阵表合并为单一数据源

## 四、AI / Ensemble 层优化

### 4.1 🟠 评估闭环：评分结果自动反馈权重

**现状**：`evaluation.py` 计算了每个 AI 模型的 helped/hurt 统计，但结果未反馈到 Ensemble 权重中。这是一个**开环系统**——有评估但无自我修正。

**问题**：
- 表现差和表现好的 AI 模型获得相同权重
- Ensemble 权重完全静态，不会根据历史准确率、数据可用性或置信度动态调整
- 模型退化或 API 质量下降时无自动降级

**优化方案**：
1. **动态权重调整**：基于滚动窗口（最近 N 场）的 Brier 分数，动态调整 AI 内部权重
   ```python
   # 伪代码
   ai_weight_i = base_weight_i * (1 / (recent_brier_i + epsilon))
   ai_weights = normalize(ai_weights)  # 归一化
   ```
2. **退化检测与自动降级**：当某模型连续 M 场 Brier 高于基线时，自动将其权重降为 0 并告警
3. **置信度加权**：AI 返回的置信度信息（如有）应影响 Ensemble 权重
4. **自动模型推荐**：在复盘页面增加"推荐使用哪个模型版本"的建议

---

### 4.2 🟡 AI v2 独立 prompt 模型激活与验证

**现状**：YAML 中 4 个 v2 模型全部 `enabled: false`，`include_in_ensemble: false`。

**问题**：
- v2 的设计初衷（避免 AI 锚定系统概率、提供独立信号）完全无法生效
- 当前所有 AI 预测都使用 v1 prompt（直接暴露 raw probs/xG），AI 可能只是鹦鹉学舌
- `analyze_prompt_independence()` 函数已实现但从未被调用

**优化方案**：
1. **A/B 测试**：先启用 1-2 个 v2 模型作为 shadow，与 v1 并行运行但不参与 Ensemble
2. **独立性验证**：调用 `analyze_prompt_independence()` 检测 v1/v2 输出与系统预测的偏差度
3. **分阶段接入**：如果 v2 确实提供了独立信号且 Brier 不劣于 v1，逐步将其纳入 Ensemble

---

### 4.3 🟡 AI 解析器收紧验证

**现状**：`parser.py` 允许单概率值最高到 1.5，总和在 [0.80, 1.20] 范围内即接受。

**问题**：
- 概率公理要求 [0,1] 且和=1，0.80-1.20 的范围过于宽容
- 虽然有自动归一化，但宽松验证可能掩盖模型输出质量问题
- 归一化后概率的含义可能偏离模型原始意图

**优化方案**：
1. 收紧验证范围：单值 [0, 1]，总和 [0.90, 1.10]
2. 超出范围时记录 warning 级别日志
3. 累计 warning 次数超过阈值时在复盘页面展示

---

### 4.4 🟢 并发安全与去重

**现状**：批量 AI 预测的 DB 写入缺乏事务保护，去重仅靠查询。

**优化方案**：
1. 增加 `match_id + model_version` 联合唯一索引
2. 写入使用 `INSERT OR REPLACE` / `UPSERT` 语义
3. 批量写入使用事务包裹

## 六、工程架构层优化

### 6.1 🟡 评分不区分比赛难度

**现状**：所有比赛使用相同的 Brier/log loss 评估。

**问题**：预测 0.9 热门胜场和预测 0.4 均势场的难度完全不同。Brier 对 0.9 预测的惩罚力度天然大于 0.5 预测。

**优化方案**：
1. 增加"难度分层"：按赛前预测的 max_prob 分为"热门"（>0.6）、"均势"（0.4-0.6）、"冷门"（<0.4）
2. 分层计算 Brier，避免被热门场的低 Brier 掩盖均势场的高 Brier
3. 增加"信息分"（information score）：模型比均匀预测（0.33/0.33/0.33）多提供了多少信息

---

### 6.2 🟡 职责分散与代码重复

| 问题 | 位置 |
|------|------|
| 市场融合在 3 处涉及 | `model_configs.yaml`（参数）、`poisson.py`（空壳 `blend_with_market()`）、`recompute.py`（实际逻辑） |
| 赛程定义重复 | `bracket.py` 和 `qualification.py` 各自维护 32 强对阵表 |
| 循环导入风险 | `service.py` 函数级导入 `team_profiles.service` 和 `services.scoring` |

**优化方案**：
1. 市场融合统一到 `poisson.py` 的 `blend_with_market()` 方法，删除 `recompute.py` 中的冗余逻辑
2. 赛程定义合并为单一数据源
3. 重构模块导入关系，消除循环依赖

---

### 6.3 🟢 模拟迭代次数自适应

**现状**：Monte Carlo 模拟默认 10K 次，固定不变。

**优化方案**：
1. 对概率接近 50% 的场景（出线概率在 40-60%），自动提升到 50K 次以降低标准误差
2. 记录每次模拟的样本量和标准误差，供前端展示

## 七、优化路线图建议

### Phase 1：数据基础（1-2 周）

- [ ] Team Profile 数据从 mock 切换到真实历史数据
- [ ] 市场数据源增加 Bet365/Pinnacle 备选
- [ ] 明确比赛结果自动同步机制

### Phase 2：模型校准（2-3 周）

- [ ] 实现 Poisson overdispersion（Negative Binomial）
- [ ] 用 2014/2018/2022 世界杯数据做参数校准
- [ ] 淘汰赛模拟参数校准
- [ ] 启用 v2 prompt 模型做 A/B 测试

### Phase 3：闭环与可视化（2-3 周）

- [ ] 实现 Ensemble 动态权重（评分结果反馈）
- [ ] 前端引入 Recharts/ECharts 图表库

## 八、关键发现摘要

### 🏆 做得好的地方

1. **赛前快照与评分口径**：24h 锁定→开球前快照的演进路径清晰，评分不污染赛前的原则执行到位
2. **模型版本隔离**：baseline / profile / shadow / AI / ensemble 各自独立版本，互不污染
3. **AI 状态真实性**：区分 ready/disabled_no_key/provider_error，不假装 ready
4. **配置驱动**：model_configs.yaml 和 ai_models.yaml 的配置化设计灵活可扩展
5. **工作流自动化**：daily-open / pre-match / post-match 工作流链路完整

### ⚠️ 最大的 3 个风险

1. **Team Profile 完全依赖假数据**——画像调整规则在假数据上验证无意义
2. **Poisson 模型无法表达 overdispersion**——导致平局概率系统性偏低，依赖 draw_boost 后处理弥补
3. **评估与决策脱节（开环系统）**——有评分但无自动反馈，模型退化时无告警
